2125A - Difficult Contest

Idea: FelixArg

Tutorial

2125A - Difficult Contest

Notice that in the strings "FFT" and "NTT", the letter "T" is at the end. From this, we can come up with a solution: place all the letters "T" before all the letters "F" and "N".

We can do this directly in one pass through the string, or we can use a slightly more clever approach: let's sort all the letters in non-increasing order, then "T" will come before "F" and "N" in the sorted string, since "T" has a higher position in the alphabet.

Asymptotic complexity: O(n) or O(nlogn)

Solution (awoo)
#include<bits/stdc++.h>
using namespace std;
 
int main(){
    int tc;
    cin >> tc;
    while (tc--){
        string s;
        cin >> s;
        sort(s.begin(), s.end());
        reverse(s.begin(), s.end());
        cout << s << '\n';
    }
}

2125B - Left and Down

Idea: BledDest

Tutorial

2125B - Left and Down

In all the examples from the problem statement, the answer is no more than 2, and indeed, it is always possible to reach the point (0,0) by using the pair (0,1) a times and the pair (1,0) b times, with a total cost of 2.

So, we need to check if it is possible to reach it at a cost of 1. Since the cost is 1, we need to find a pair (dx,dy) such that dxk, dyk, and it is possible to reach the cell (0,0) by consistently applying this pair. This means there exists an integer t such that (a=tdx,b=tdy), and accordingly, t is a divisor of both a and b.

If there exists such t that atk and btk, then we can use the pair (at,bt) t times at a cost of 1. Theoretically, we could iterate through all divisors of a and b and check these conditions for each. However, the limits do not allow us to do this in either O(a) or O(a). Note that t appears in the denominator of the fraction, which means we need to choose it as great as possible. Therefore, it is enough to check the only value of tthe greatest common divisor of a and b. This can already be found by using the built-in function gcd(a,b).

Solution (ifive)
from math import gcd
for _ in range(int(input())):
    a, b, k = map(int, input().split())
    g = gcd(a, b)
    print(1 if a // g <= k and b // g <= k else 2)

2125C - Count Good Numbers

Idea: BledDest

Tutorial

2125C - Count Good Numbers

To begin with, it is worth mentioning that if you need to count the number of certain integers in the interval [l,r], it is almost always more convenient to do it this way: count the number of such integers in [0,r+1) (or in [1,r+1)), and then subtract the count in [0,l) (or in [1,l)). Counting the number of integers on a prefix is usually significantly easier than in an interval.

Let's slightly modify the definition of a good number. If all prime factors of an integer consist of at least two digits, then it is not divisible by any of the first four prime numbers (2,3,5,7). Similarly, if a number is not good, then it is divisible by at least one of these four numbers.

There are at least two options for solving the problem: through the periodicity of good numbers and through the inclusion-exclusion formula.

A number is good if it is not divisible by 2,3,5,7. This means that it is sufficient to know the remainder of the number when divided by 2, 3, 5, and 7 to determine whether it will be good. This can be determined by the remainder when divided by L=2357=210: if x is good, then x+L is also good, and vice versa.

This means that if we need to count the number of good integers from 0 to k1, we can calculate the number of complete segments of length L in [0,k1] (this is kL), multiply this by the number of good integers in [0,L1] (this can be naively computed in O(L)), and then handle the part of the last segment with kmodL extra integers. This way, we will get a solution in O(L) per query.

The solution with the inclusion-exclusion formula works as follows: let's understand how to count the number of integers that are coprime to 210 in the interval [1,k] (if a number is coprime to 210, then it is good, as it is not divisible by any of the primes we are interested in). We will form four sets S2,S3,S5,S7, where Si contains all integers from 1 to k that are divisible by i. Using the inclusion-exclusion formula, we can find the union of these four sets as follows:

|S2|+|S3|+|S5|+|S7||S2S3||S2S5|+|S2S3S5|+|S2S3S7|+|S2S3S5S7|

How do we calculate |Si|? It is simply ki, as we need exactly the numbers i,2i,. How do we calculate the size of the intersection of several sets Si? If a number is divisible by several primes, then it is divisible by their product, so in fact the set Sp1Sp2Spm can be represented as Spi; for example, S2S3S5=S30. This way, we will obtain a solution that works in O(2P) per query, where P is the number of primes we are interested in (in this case, P=4).

Solution 1 (BledDest)
def good(x):
    return x % 2 > 0 and x % 3 > 0 and x % 5 > 0 and x % 7 > 0
 
def get_naive(x):
    ans = 0
    for i in range(x):
        if good(i):
            ans += 1
    return ans
 
def get(r):
    return (r // 210) * get_naive(210) + get_naive(r % 210)
 
t = int(input())
for i in range(t):
    l, r = map(int, input().split())
    print(get(r+1) - get(l))

2125D - Segments Covering

Idea: BledDest

Tutorial

2125D - Segments Covering

There are two ways to solve this problem, both based on dynamic programming.

The first approach is to calculate a dynamic programming of the form dpx the probability that all cells from 1 to x are covered exactly once. The base case for the dynamic programming is dp0=1. The transitions are as follows: to compute dpx, we need to iterate over all segments i that end at x (i.e., segments i such that ri=x; these can be pre-stored in a cell numbered x), and for each such segment, we increase dpx by the probability that the following three conditions are met:

  • we covered everything up to li1 exactly once (this is simply dpli1);
  • segment i exists (this is piqi);
  • we did not cover any cells from li to ri with other segments. The probability of this can be calculated as follows: all segments covering cells from li to ri are divided into two groups: segments that start before li but end in [li,m], and segments that start in [li,ri]. For segments of the first group, we already know that they do not exist, since all cells up to li1 are covered exactly once (otherwise, cell li1 would be covered by at least two segments); for segments of the second group, we do not know anything yet, so we need to take the probability that none of them exist. To quickly calculate such a probability, we can form an array where the j-th cell contains the probability that no segment starting at j exists (i.e., the product qkpkqk for all segments where lk=j), and take the product over that segment of the array (this can be done using prefix products, similar to how sums over segments are calculated using prefix sums).

The second solution is similar to the first but is significantly simpler to implement. We will consider the dynamic programming array dpx the probability that all cells from 1 to x are covered exactly once, and none of the other cells are covered. The base case for this dynamic programming is: dp0 is equal to the probability that no segment exists (i.e., the product qipiqi for all segments). The transitions are as follows: to compute dpx, we need to iterate over all segments i that end at x, and for each of them, increase dpx by the probability that all cells from 1 to li1 are covered exactly once, and that segment i exists. At the first glance, this is equal to dpli1piqi; however, this expression needs to be divided by qipiqi, since in dpli1 it is assumed that this segment does not exist. This solution does not require any prefix products (unlike the first solution), so it is easier to code.

Finally, let's talk about how to work with fractions modulo 998244353 (I will refer to this number as M). This is a prime number, so every integer from 1 to M1 has an inverse element modulo this number. To find it, one can either use the extended Euclidean algorithm, or raise the number to the power of M2 using binary exponentiation (for a prime M and each number x from 1 to M1, the equality xM1modM=1 holds, so the inverse element for x will be xM2). Each fraction of the form xy can simply be treated as the number xy1modM, and when adding fractions, one can simply add the corresponding numbers without worrying about reducing fractions, finding a common denominator, and similar issues. In the first solution using prefix products, we will have to divide by some precomputed number to obtain the product over a segment for this, it is sufficient to find its inverse element and multiply by it.

Solution 1 (Neon)
#include <bits/stdc++.h>
 
using namespace std;
 
using pt = pair<int, int>;
 
const int MOD = 998244353;
 
int add(int x, int y) {
   x += y;
   if (x >= MOD) x -= MOD;
   if (x < 0) x += MOD;
   return x;
}
 
int mul(int x, int y) {
  return x * 1LL * y % MOD;
}
 
int binpow(int x, int y) {
  int z = 1;
  while (y) {
    if (y & 1) z = mul(z, x);
    x = mul(x, x);
    y >>= 1;
  }
  return z;
}
 
int divide(int x, int y) {
  return mul(x, binpow(y, MOD - 2));
}
 
int main() {
  ios::sync_with_stdio(false); cin.tie(0);
  int n, m;
  cin >> n >> m;
  vector<vector<pt>> a(m + 1);
  for (int i = 0; i < n; ++i) {
    int l, r, p, q;
    cin >> l >> r >> p >> q;
    a[r].emplace_back(l - 1, divide(p, q));
  }
  vector<int> pr(m + 1);
  pr[0] = 1;
  for (int i = 1; i <= m; ++i) {
    int cur = 1;
    for (auto [_, p] : a[i]) cur = mul(cur, add(1, -p));
    pr[i] = mul(pr[i - 1], cur);
  }
  vector<int> dp(m + 1);
  dp[0] = 1;
  for (int i = 1; i <= m; ++i) {
    for (auto [l, p] : a[i]) { 
      int cur = divide(pr[i], pr[l]);
      cur = divide(cur, add(1, -p));
      cur = mul(cur, p);
      dp[i] = add(dp[i], mul(dp[l], cur));
    }
  }
  cout << dp[m] << endl;  
}

2125E - Sets of Complementary Sums

Idea: FelixArg

Tutorial

2125E - Sets of Complementary Sums

Let's first understand the following: what sets Q can be obtained from some arrays a using the operations described in the problem statement.

To do this, let's take an arbitrary set Q and try to reconstruct any array from it. We know that the initial sum s is at least max(Q)+1, because otherwise there would be a number 0 in the array.

Let's try to reconstruct some array with a sum exactly equal to max(Q)+1. We will create an empty array a and then iterate through the elements of the set Q, adding to a the numbers max(Q)+1x, where xQ. Okay, now let's calculate the sum of all elements in our a, and there can be three scenarios:

  1. The sum equals the initial max(Q)+1; in this case, nothing needs to be done.
  2. The sum is less than max(Q)+1; in this case, we need to add several copies of some number from those currently in a.
  3. The sum is greater than max(Q)+1; in this case, the original set Q cannot be obtained using the operations described in the problem statement.

For point two, it's easy: we notice that our a contains the number 1, so we can always make the sum of a equal to max(Q)+1 simply by adding several ones.

Now let's consider the third point: we were unable to create the array, assuming its sum is max(Q)+1, but could the initial sum have been greater? Let's try to assume that the initial sum was max(Q)+2; however, we will again arrive at this third point. Let's try max(Q)+3, and it won't work either, nor will any subsequent value. Why is that?

Let's express the sum of our current a under the assumption that it is max(Q)+1 in a more mathematical form:

yQ(max(Q)+1y)=(max(Q)+1)|Q|yQy

Now let's write the sum if the assumption was max(Q)+2:

(max(Q)+2)|Q|yQy

Subtract one from the other:

(max(Q)+2)|Q|yQy((max(Q)+1)|Q|yQy)=(max(Q)+2)|Q|(max(Q)+1)|Q|=|Q|

It turns out that when we increased our assumption by 1, the sum increased by the size of the set Q, and since the size of Q is always at least one, we will never achieve equality between our assumption and the actual sum.

From this, we can conclude that for any set Q, a necessary and sufficient condition for it to be a set of complementary sums is:

(max(Q)+1)|Q|yQy(max(Q)+1)

or in a slightly different form:

(max(Q)+1)(|Q|1)yQy0(1)

Now let's finally solve the problem itself. We would like to count such sets for which the above condition holds, and where |Q|=n and max(Q)x.

To do this, we start with the set {1,2,...,n}; from it, we can obtain other sets using one simple operation: adding 1 to the suffix of the elements of the set.

For each such operation, we define f(i) as the change in the sum on the left side of condition (1) if we added 1 to the suffix of length i. We can easily calculate all f(i); they are equal to:

f(1)=n2 f(2)=n3 f(3)=n4 f(n1)=0 f(n)=1

Notice that the only operation that decreases the left side of condition (1) is f(n). This operation adds 1 to all elements of the set Q.

So, we have the set {1,2,...,n}; it may violate condition (1), and the only thing we can do is apply several operations of adding 1 to all elements to satisfy equality in (1). Let the number of such operations be st.

Now let's perform some operations in such a way that this equality is maintained. Suppose we added 1 to the suffix of length j; to maintain equality, we need to perform f(j) more additions of 1 to all of Q, while the maximum in Q increases by f(j)+1.

All these considerations lead us to such an implementation: let dp[i] be the number of different sets of complementary sums such that the maximum in these sets equals i and equality in condition (1) holds; the initial state is dp[st+n]=1.

We will recalculate this dynamic programming as in the knapsack problem. We can represent the operations of adding to the suffix of length j as an item with weight f(j)+1 that we put into the knapsack:

for (int i = 0; i < n - 1; i++){
    for (int j = 0; j < x; j++){
        if (j + (i + 1) <= x){
            dp[j + (i + 1)] = (dp[j + (i + 1)] + dp[j]) % MOD;
        }
    }
}

Thus, we will count the sets of complementary sums for which the inequality (1) holds; all others can be obtained from those counted by performing +1 operations on the entire set. Specifically, we need to make one more pass over the dynamic programming, effectively turning it into a prefix sum array:

for (int i = 0; i < x; i++){
    dp[i + 1] = (dp[i + 1] + dp[i]) % MOD;
}

After this, the answer will simply be the sum of the entire array dp.

There remains one problem: the current solution works in O(nx) and does not fit within the time limits.

To optimize our solution, let's take a closer look at this inequality (1). If the sum of all elements in Q is large, then theoretically max(Q) is large and |Q| is also large. What is the maximum possible sum of the elements in Q?

If we take Q={max(Q)n+1,max(Q)n+2,,max(Q)}, we can easily derive the formula:

maxyQy=max(Q)|Q||Q|(|Q|1)2

Substituting into condition (1):

(max(Q)+1)(|Q|1)max(Q)|Q|+|Q|(|Q|1)20

From here, we can express the relationship:

max(Q)|Q|(|Q|1)2+|Q|1

From this, we see that the maximum in Q depends on the square of |Q|, which means that for large n, the answer is always 0. In other cases, we can compute our dynamic programming in O(xx).

Solution (FelixArg)
#include<bits/stdc++.h>
using namespace std;
 
#define int long long
 
const int MOD = 998'244'353;
 
void solve(){
    int n, x;
    cin >> n >> x;
    // need sum(mx + 1 - ai) <= mx + 1
    // start with 1, 2, 3, ... n
    // min sum + 1 is n + 1
 
    if (n == 1){
        cout << x << '\n';
        return;
    }
 
    int s_cur = n * (n + 1) / 2;
    int need = s_cur - (n + 1);
 
    // leads to n is O(sqrt(s))
    if (n + need > x){
        cout << 0 << '\n';
        return;
    }
 
    // so we can do dp[sum] knapsack
 
    vector<int> dp(x + 1);
    dp[0] = 1;
 
    // for -1 and 0
    for (int j = 0; j < 2; j++){
        for (int i = 0; i < x; i++){
            dp[i + 1] = (dp[i + 1] + dp[i]) % MOD;
        }
    }
 
    for (int i = 1; i <= n - 2; i++){
        for (int j = 0; j < x; j++){
            // increase sum by i -> mx increase by 1 + i
            if (j + i + 1 <= x){
                dp[j + i + 1] = (dp[j + i + 1] + dp[j]) % MOD;
            }
        }
    }
 
    int ans = 0;
    for (int i = 0; i <= x; i++){
        int cur = n + need + i;
        if (cur > x){
            break;
        }
        ans = (ans + dp[i]) % MOD;
    }
 
    cout << ans << '\n';
}
 
signed main()
{
#ifdef FELIX
    auto _clock_start = chrono::high_resolution_clock::now();
#endif
    ios_base::sync_with_stdio(false);
    cin.tie(0);
    cout.tie(0);
 
    int t = 1;
    cin >> t;
    while(t--){
        solve();
    }
 
#ifdef FELIX
    cerr << "Executed in " << chrono::duration_cast<chrono::milliseconds>(
        chrono::high_resolution_clock::now()
            - _clock_start).count() << "ms." << endl;
#endif
    return 0;
}

2125F - Timofey and Docker

Idea: FelixArg

Tutorial

2125F - Timofey and Docker

Hint: The most challenging case is when we need to increase the number of occurrences of the word "docker". This can be considered as a dynamic programming problem of the form dp[i][cnt] in O(|s|2). Is there a way to optimize it?

Let's calculate the maximum number of people that can be achieved, let's denote this as mx. Next, we will count the number of occurrences of the string t="docker" in s, let's denote this as cnt. Here we can notice the following: the occurrences of the string t do not overlap since t consists of different letters.

Then, we will look at all the possible occurrences of the string t="docker" in s that achieve mx. Formally, we can divide them into 2 groups: cnt and >cnt.

In the first case, we need to change some letters of the already existing occurrences, then we can update the answer as ans=min(ans,cnti), where i indicates that we need exactly i occurrences.

In the second case, we need to apply dynamic programming of the form dp[x][y]the minimum number of characters that need to be replaced in the prefix s of length x so that the number of occurrences is exactly y. This dynamic programming can be computed in O(|s|i). It is sufficient to compute this dynamic programming only once for such i that achieves mx with i occurrences, and i is the first such that it is cnt.

Now we need to optimize the dynamic programming. We can replace it with the fact that the function g(x)=dp[|s|][x] is non-decreasing and, in addition, convex, so we can apply the "aliens trick" to find dp[|s|][i].

Let's recall how to determine a convex function. It is sufficient for the following condition to hold:

g(x1)g(x)g(x)g(x+1)

For our g(x), this condition holds, and it is quite intuitive that adding each new occurrence of the string "docker" in s becomes more expensive (i.e., it requires at least as many character replacements for each new occurrence of "docker" in s), but let's prove this.

Our quadratic dynamic programming has the form:

dp[i][j]=mink<idp[k1][j1]+w(k,i)

Here, w(k,i) is defined as follows:

  • w(k,i)=, if ik+1<6;
  • w(k,i)= the minimum number of character replacements in some subsegment of the segment (k,i) to turn it into "docker".

Note that the quadrangle inequality holds:

w(a,c)+w(b,d)w(a,d)+w(b,c)

for 1abcd|s|.

Why does it hold? Suppose we have chosen some optimal subsegment of length 6 for w(a,d), then this segment could have been chosen either in w(a,c) or in w(b,d). At the same time, the segment for w(b,c) can also be chosen in both w(a,c) and w(b,d).

The fulfillment of the quadrangle inequality is sufficient to state that the function is convex. The proof of this fact is quite extensive and will not be provided here; however, you can search for it on the internet.

Final asymptotics: O(n+|s|logC)

            if (dx[i] != mxcnt) continue;
            if (i <= cur){
                ans = min(ans, cur - i);
                continue;
            }
            k = i;
            break;
        }

            cout << ans << '\n';
            continue;
        }
 
        auto calc = [&](int d){
            vector<pair<long long, int>> dp(n + 1, {(long long)1e18, -1});
            dp[0] = {0, 0};
            forn(i, n){
                dp[i + 1] = min(dp[i + 1], dp[i]);
                if (i + 6 <= n)
                    dp[i + 6] = min(dp[i + 6], {dp[i].first + dif[i] + d, dp[i].second - 1});
            }
            return dp[n];
        };
 
        int l = -INF, r = 0, sv = INF;
        while (l <= r){
            int m = (l + r) / 2;
            auto [val, cnt] = calc(m);
            if (-cnt >= k){
                sv = val - k * 1ll * m;
                l = m + 1;
            }
            else{
                r = m - 1;
            }
        }
        ans = min(ans, sv);
 
        cout << ans << '\n';
    } 
}

Slightly different derivation of the sufficient and necessary condition for question E. Consider any (sorted) output set x = {x(1),x(2),...,x(n)} that we want to make a input set for for. Consider the difference array {d(2),d(3),..d(n)} where d(i) = x(i)-x(i-1) we know there must be some term z after which the following terms come, z+d(n), z+d(n)+d(n-1), z+d(n)+d(n-1)+d(n-2).., in the input set. Thus let us only have those terms for now. Note that it is optimal to make z=1, since any higher z would only make the input sum higher raising all our output terms by a constant, which we can just do instead by adding more 1 terms should we need to. Now we have the input array {1,1+d(n),1+d(n)+d(n-1),...,1+(sum of all d(i)}. Lets find the sum of this array, note that 1 adds to n terms, (x(n)-x(n-1)) adds to n-1 terms, (x(n-1)-x(n-2)) adds to n-3 terms, etc. It is not hard to see that each x term will be subtracted once except for x(n), thus the final sum is just n+x(n)*(n-1)-(sum of all x(i) s.t i<n). Now, from the construction, we know that this corresponds to an output array that is off by the original set x by some constant, i.e it has the same difference array when sorted. The only question now is, is the highest term, and thus all the terms, small enough? Remember that too small is ok since we can raise the sum, but too big is not ok. We know that the largest term will be equal to the sum-1, since the lowest term is 1, so we know it is just n+x(n)*(n-1)-(sum of all x(i) s.t i<n)-1 and this must be less than or equal to x(n). Thus we have n+x(n)*(n-1)-(sum of all x(i) s.t i<n) 1 <= x(n). Rearranging this we get x(n)*(n-1)+(n-1) (sum of all x(i) s.t i<n) <= x(n) and then we get (x(n)+1)*(n-1) (sum of all x(i)) <= 0, exactly the condition found in the editorial.

In the editorial for E, is dp[st]dp[st] really equal to 1? From what I understand we should have dp[st+n]=1dp[st+n]=1 instead (with dp[st]=0dp[st]=0).

dp[i]dp[i] is defined as "the number of different sets of complementary sums such that the maximum in these sets equals ii and equality in condition (1) holds". As stst is defined as the number of additions of ones to all of the elements of the set {1,2,,n1,2,,n} needed to make it satisfy the condition (1), it can be zero (with n=2n=2). With n=2n=2, dp[0]1dp[0]1, because every set of complementary sums with more than one element* has only positive values inside, so the maximum of those values can't be zero. On the other hand, dp[st+n]=1dp[st+n]=1 makes sense -- the set {1,2,,n1,2,,n} after the additions will have its maximum equal to n+stn+st, thus dp[n+st]=1dp[n+st]=1.

*in the editorial it is not specified that in the dpdp we only care about sets of size nn, but I don't see how that wouldn't be true.
